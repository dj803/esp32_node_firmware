## What this tells us about #46

This is the first fleet-wide outage observed with v0.4.23 deployed
(which has sub-G `restart_cause`, sub-B `last_restart_reasons`,
sub-F WDT 12s, jittered backoff, and `mqtt_disconnects` counter). The
data validates the assumption that v0.4.23's heartbeat fields would
make outage-class events visible — `mqtt_disconnects=1` on Delta and
Foxtrot's surviving heartbeats is exactly the signal #55 was designed
to produce.

The split — 2 devices clean reconnect, 1 int_wdt, 3 not-yet-back — is
informative but not yet decided as a pattern. The not-yet-back devices
may simply be in a longer reconnect-backoff cycle (`MQTT_RECONNECT_MAX_MS=60000`
+ jitter), in which case they'll come back over the next ~5 min. If
they don't, that's a stronger signal (panic loop, NVS corruption, or
a permanent network-config drift). Worth checking again next time the
operator engages.

## How v0.4.24's pending changes would have affected this

If the fleet had been on v0.4.24 (currently code-on-master, NOT tagged):

- **Sub-C (time-based MQTT_RESTART_THRESHOLD).** Alpha's int_wdt
  doesn't tell us whether it would have escalated to the 10 min
  unrecoverable trigger first. The watchdog firing came earlier in
  the failure mode, not later. Sub-C's value is for slower-decay
  scenarios where the count threshold under-fires.
- **Sub-D (restart-loop AP fallback).** If Alpha had re-int_wdt'd 3
  times in a row with the same restart_cause, sub-D would route to
  AP_MODE on the 4th boot. Today's single int_wdt boot does NOT
  trip sub-D (different boot_reason — int_wdt is hardware not
  software). Sub-D specifically guards `mqtt_unrecoverable` software
  restart loops.
- **Sub-I (/daily-health categorisation).** Alpha's int_wdt would
  have surfaced as a RED entry in the new fleet-rollup row. With
  v0.4.23's lumped-together categorisation operators would see "1
  abnormal" without the WDT-vs-software distinction.

So the new code is partially relevant but not load-bearing for this
specific failure mode — int_wdt is faster than any of the 10-min/3-loop
thresholds. The sub-F WDT bump 5s → 12s (already in v0.4.23) is the
relevant prior change; the int_wdt fired anyway, suggesting whatever
blocking site triggered it stayed blocked for > 12 s.
