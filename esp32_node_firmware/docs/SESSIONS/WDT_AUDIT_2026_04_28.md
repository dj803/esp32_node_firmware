# Task-watchdog audit (#29)

Read-only sweep of every blocking I/O site in the firmware against the
TWDT subscription model. Conducted 2026-04-28 against v0.4.22-source +
the v0.4.23 batch (#55 + #76 sub-B/F/H + this audit).

## TWDT configuration

Per `sdkconfig.defaults`:

| Setting | Value | Notes |
|---|---|---|
| `CONFIG_ESP_TASK_WDT_EN` | y | TWDT compiled in |
| `CONFIG_ESP_TASK_WDT_INIT` | y | Auto-init at boot |
| `CONFIG_ESP_TASK_WDT_PANIC` | y | Trip → panic (not just LOG_E) |
| `CONFIG_ESP_TASK_WDT_TIMEOUT_S` | **12** (was 5 pre-v0.4.23) | #76 sub-F bump |
| `CONFIG_ESP_TASK_WDT_CHECK_IDLE_TASK_CPU0` | y | Idle task on Core 0 must run |
| `CONFIG_ESP_TASK_WDT_CHECK_IDLE_TASK_CPU1` | (unset) | Core 1 idle NOT watched |
| `CONFIG_INT_WDT` | y | Interrupt watchdog enabled |
| `CONFIG_INT_WDT_TIMEOUT_MS` | 300 | int_wdt fires if ISR runs > 300 ms |
| `CONFIG_INT_WDT_CHECK_CPU1` | y | Both cores watched for ISR storm |

Default subscribed tasks at boot: idle Core 0. Other tasks subscribe
explicitly via `esp_task_wdt_add(NULL)` from inside the task.

## Blocking sites — categorised by risk

### 1. Explicitly TWDT-subscribed (safe — these self-feed)

| Site | File:line | Notes |
|---|---|---|
| AP-portal HTTPS keygen | `ap_portal.h:179-212` | Subscribes loopTask, calls `esp_task_wdt_reset()` after each long-running phase. v0.3.37 fix; no panics observed since. |
| OTA download (mbedTLS, HTTPS) | `ota.h` (handled inside ESP32-OTA-Pull library) | Library does its own watchdog feeding. |

### 2. Synchronous network I/O — explicit timeouts

| Site | File:line | Block bound | Risk |
|---|---|---|---|
| `WiFiClient.connect` (broker probe, #67/#78 v0.4.16) | `mqtt_client.h:_mqttBrokerProbe` | 1500 ms timeout | LOW. New 12 s WDT comfortably absorbs. |
| `WiFiClient` operations during broker discovery | `broker_discovery.h:135` | bounded by per-attempt timeout | LOW |
| HTTPClient OTA manifest fetch | `ota.h::otaCheckNow` | Default HTTPClient timeout (~15 s) | MEDIUM — exceeds 12 s WDT in worst case. otaCheckNow happens on loopTask which IS TWDT-subscribed during OTA via ESP32-OTA-Pull's internal hooks. |

### 3. ESP-NOW response wait loops

| Site | File:line | Pattern |
|---|---|---|
| Bootstrap response wait | `espnow_bootstrap.h:278,294,385,535,614,689` | `while (cond && millis() < deadline) delay(10);` |
| Responder OTA-URL/broker wait | `espnow_responder.h:414,573` | Same pattern |

All use `delay(10)` inside a bounded loop. `delay()` yields to FreeRTOS so
the idle task on Core 0 can feed the WDT. Bounded deadlines (typically
500-6500 ms) keep total wait < WDT timeout. **SAFE**.

### 4. Bare `delay()` calls

| Site | File:line | Length | Notes |
|---|---|---|---|
| AP portal post-save settle | `ap_portal.h:780` | 2000 ms | One-shot before `ESP.restart()`. Safe — yields to idle. |
| AP portal halt-on-error | `ap_portal.h:850` | `while (true) delay(1000)` | Intentional dead-stop; only path is reboot via RTC WDT. |
| AP portal HTTPS-drain | `ap_portal.h:887,899,918` | 100-200 ms each | One-shot, idle yields. |
| Broker discovery yields | `broker_discovery.h:456,480` | 5-30 ms | Yields to async_tcp/lwIP between probe attempts. |
| Coredump publish | `coredump_publish.h:175` | 150 ms | Drain wait; one-shot at boot. |
| ESP-NOW bootstrap settles | `espnow_bootstrap.h:251,502,566,570` | 50-100 ms | Channel-switch settle / radio re-init. |
| Pre-restart drain | `mqtt_client.h:945,950` | 10 ms × ~50, then 50 ms | Bounded `while`; total drain < 600 ms. Safe. |

### 5. Direct hardware operations (no yield)

| Site | File:line | Risk |
|---|---|---|
| ws2812 `FastLED.show()` | `ws2812.h` (FastLED internal) | RMT peripheral DMA, ~1 ms for 8 LEDs. Cheap. |
| RFID SPI transactions | `rfid.h` | MFRC522 transactions ~5-10 ms each. Bounded. |
| Hall ADC reads | `hall.h` (v0.5.0) | 8 sample averages, < 1 ms total. |

## Findings

1. **All non-TWDT-subscribed sites stay under 600 ms** of wall-clock
   block, thanks to the `delay(10)` loop pattern and short `delay(N<=200)`
   one-shots. With the new 12-second TWDT (was 5), every site has at
   least a 20× margin.

2. **HTTPClient OTA manifest fetch** is the closest to the TWDT edge.
   ESP32-OTA-Pull internally manages WDT subscription for the actual
   download; the manifest fetch itself is a single GET that's bounded
   by HTTPClient's default 15 s timeout. Pre-v0.4.23 with TWDT=5 s
   this was a known risk; with TWDT=12 s the worst-case 15 s manifest
   fetch could STILL trip if the pull library doesn't subscribe in
   time. **Recommend**: explicitly call `esp_task_wdt_reset()` in
   the otaCheckNow path before `pull.CheckForOTAUpdate()`. Filing as
   a sub-followup of #29.

3. **No bare-spinwait anywhere** — every `while` block has a
   `millis() < deadline` guard. v0.3.x cleanup paid off.

4. **Core 1 idle is NOT TWDT-watched** (CONFIG_ESP_TASK_WDT_CHECK_IDLE_TASK_CPU1
   unset). LED + RFID + ws2812 tasks pin to Core 1; if any locks the core
   without yielding, only the int_wdt (300 ms ISR-block) catches it —
   not the longer task_wdt. **Recommend**: enable
   CONFIG_ESP_TASK_WDT_CHECK_IDLE_TASK_CPU1 in a future sdkconfig
   tightening pass; bundle with #76 sub-D (cool-off counter).

## Status

This document IS the audit deliverable for #29. No code changes were
required — the existing pattern of bounded `delay(10)` loops + explicit
TWDT subscriptions in known-long blocks is sufficient. Two
recommendations file separately as audit-followups (HTTPClient
manifest-fetch WDT reset; CPU1 idle watch).

#29 → RESOLVED 2026-04-28 in v0.4.23 batch.
