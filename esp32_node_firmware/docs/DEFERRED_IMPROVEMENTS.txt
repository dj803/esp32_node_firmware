ESP32 Node Firmware + Node-RED — Deferred Improvements
======================================================
Generated: 2026-04-21

These items were surfaced by the v0.3.03 planning audit but intentionally left
out of the current release window. Keep this file as the backlog; revisit before
the next major version bump.

────────────────────────────────────────────────────────────────────────────────
FIRMWARE — SECURITY
────────────────────────────────────────────────────────────────────────────────

1. (shipped in v0.3.07) CSRF tokens on HTTP portals
   include/ap_portal.h: generateCsrfToken() uses esp_random() for 128-bit
   tokens. GET /  and GET /settings generate fresh tokens, embed them in
   hidden form fields, and set SameSite=Strict; HttpOnly cookies.
   POST /save and POST /settings verify the submitted field matches the
   server-stored token (single-use — invalidated after a successful check)
   and return HTTP 403 on failure.

2. (moved to WONT_DO.txt — decision not to change the hardcoded AP password)

3. HTTPS for provisioning portals (MEDIUM)
   Problem: Plain HTTP portals allow eavesdropping of entered credentials on
            shared LAN.
   Fix:    Use ESP-IDF's httpsd component with a self-signed cert; accept the
           UX cost of browser warnings.

────────────────────────────────────────────────────────────────────────────────
FIRMWARE — PERFORMANCE / BOOT TIME
────────────────────────────────────────────────────────────────────────────────

4. Non-blocking ESP-NOW channel scan (HIGH)
   File:   include/espnow_bootstrap.h lines 289, 304, 388, 461, 540, 607
   Problem: Blocking `while (!_response && millis() < deadline) delay(10)` on
            every channel. Full 1-13 sweep ≈ 6.5 s worst case, blocking setup().
   Fix:    Convert to a state-machine / FreeRTOS-task pattern driven from loop().

5. Non-blocking broker port scan (HIGH)
   File:   include/broker_discovery.h line 348
   Problem: Synchronous TCP connect per host, 75 ms timeout × 254 hosts ≈ 19 s
            worst case. Blocks setup().
   Fix:    Use AsyncTCP (already in lib_deps) for parallel connects with a
            collector callback; early-exit at PORTSCAN_MAX_RESULTS.

6. (shipped in v0.3.06) Remaining blocking delay()s in setup / loop
   — loop() Wi-Fi reconnect delay(3000) replaced with millis-based non-blocking
     pattern in main.cpp. setup() retry delays carry TODO(v0.3.07) comments and
     will be removed when the boot path becomes a state machine.
   — ap_portal.h delays deferred to v0.3.11 (HTTPS refactor touches same file).

FIRMWARE — OBSERVABILITY
────────────────────────────────────────────────────────────────────────────────

7. (shipped in v0.3.06) Structured logging
   include/logging.h created. LOG_D/I/W/E macros adopted across mqtt_client.h,
   ota.h, and main.cpp. Compile-time filtered via LOG_LEVEL (default INFO).
   Remaining modules (espnow_bootstrap.h, broker_discovery.h, rfid.h, ble.h,
   ap_portal.h, espnow_responder.h, espnow_ranging.h) will adopt LOG_* when
   touched by their respective releases (v0.3.07/08/09/11).

8. (shipped in v0.3.06) Structured event codes
   include/fwevent.h created. FwEvent enum + fwEventName() lookup.
   mqttPublishStatus(FwEvent) typed overload added to mqtt_client.h.
   All mqttPublishStatus() call sites in mqtt_client.h and ota.h updated.

FIRMWARE — CORRECTNESS
────────────────────────────────────────────────────────────────────────────────

9. Consolidate module-local state flags (HIGH)
   Files:  ap_portal.h:291 (_settingsServerRunning),
           espnow_responder.h:27 (_responderActive),
           rfid.h:59 (_rfidReady)
   Problem: These flags can drift out of sync with top-level currentState.
   Fix:    Either promote each to a bit in a central RuntimeState bitmask, OR
           document that each is purely internal and expose only via accessor
           functions so misuse is impossible.

10. Per-peer rate limiting for ESP-NOW responder (HIGH)
    File:  espnow_responder.h (no rate table present today)
    Problem: A misbehaving sibling can flood credential requests, forcing
             encrypt/ECDH cycles on every request. DoS vector.
    Fix:   Per-MAC token bucket in responder — drop requests above N/minute.

11. Wire-format versioning for ESP-NOW payloads (MEDIUM)
    Files: espnow_bootstrap.h (WireBundle), espnow_responder.h (SiblingHealth)
    Problem: Struct-memcpy over ESP-NOW with no version byte. Any new field is
             a breaking fleet-wide change.
    Fix:   Add `uint8_t wire_version` as first byte of every ESP-NOW struct;
           bump and migrate deliberately.

12. Heap churn in request handlers (LOW)
    File:  ap_portal.h:94-100 (apHandleStatus JSON via String concatenation)
    Problem: Repeated String allocation/concat on every /status request.
    Fix:   Use snprintf into a stack buffer.

13. NVS namespace registry (LOW)
    Files: config.h defines 5 namespaces scattered across modules
    Problem: No single list; easy to accidentally reuse a name.
    Fix:   Consolidate into a single `enum class NvsNs` with name-lookup.

14. Expanded unit test coverage (LOW)
    File:  test/test_native/test_main.cpp (after PlatformIO migration)
    Gaps:  semverIsNewer() edge cases (0.2.15 vs 0.2.7 numeric vs
           lexicographic); credential bundle round-trip; topic path assembly
           with pathological segment values.

NODE-RED
────────────────────────────────────────────────────────────────────────────────

15. Migrate to Node-RED Projects mode (MEDIUM)
    Problem: flows.json is a 62 KB flat blob, auto-reformatted on save;
             credentials stored in plaintext in flows_cred.json; git diffs
             are unreadable.
    Fix:   Enable Projects mode; split flows into modules; separate creds.

16. Extract repeated patterns to subflows (MEDIUM)
    Problem: "MQTT in → JSON parse → table update" pattern repeats ~4 times
             (Heartbeats, BLE, ESP-NOW); "peer selection → filter → chart"
             repeats in BLE and ESP-NOW.
    Fix:   Promote each to a subflow with parameter inputs.

17. Add catch nodes / error handling (MEDIUM)
    Problem: Zero catch nodes across all 3 tabs; 24 function nodes have no
             try/catch. JSON parse failure or broker drop silently breaks
             downstream UI.
    Fix:   One catch node per tab → debug + status template;
           wrap critical function nodes in try/catch with sensible fallback.

18. Tab reorganisation (LOW)
    Problem: 3 tabs, 70 nodes. Heartbeats tab conflates status monitoring,
             file logging, and device registry — candidates for split.

19. Dashboard 2.0 authentication (LOW)
    Problem: Dashboard is open to the LAN. Fine for dev; risky if exposed to
             a wider network.
    Fix:   httpNodeAuth in settings.js with strong credentials.
