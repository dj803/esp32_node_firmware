  Early CI wasn't firing on master pushes — only release branches.

[e12804a] arduino-cli PATH — use $GITHUB_WORKSPACE/bin, not $HOME/bin
  The install.sh script drops arduino-cli under GITHUB_WORKSPACE/bin on

[c7edb71] Restore workflow to repo root after directory reorganisation
[2547dc3] Update sketch and config.h paths for versioned directory layout
  When the project moved into an esp32_node_firmware-0.1.6/ subfolder, the
  workflow needed path prefixes. This burned again in v0.3.03 (see below).

[4c59acb] Grant contents:write permission for release creation
  Default GITHUB_TOKEN is read-only on new runners; softprops/action-gh-release
  needs contents:write to create a release.

[3c46333] Fix workflow path for new repo layout (v0.3.03)
  GitHub Actions only reads workflows from the REPO ROOT `.github/workflows/`.
  A nested esp32_node_firmware/.github/workflows/build.yml is silently ignored.
  Fixed by placing the workflow at the repo root and setting
  `defaults.run.working-directory: esp32_node_firmware`.

[476b739] Fix setup-python cache + arduino-cli lib install
  setup-python@v5 now ERRORS (not warns) when cache:pip is set but there's
  no requirements.txt / pyproject.toml. Solution: drop `cache: pip`.
  Also added `arduino-cli lib update-index` explicitly before the install —
  the library index can be stale on a fresh runner.

[8810d6c] Widen parity-check tolerance to 128 KB
  v0.3.03's Arduino CLI vs PlatformIO parity check was set to 4 KB; baseline
  drift is ~34 KB because the two toolchains use different compiler revs and
  slightly different library versions. 128 KB tolerance is loose enough to
  tolerate library-version bumps and tight enough to catch an accidental
  omission of an entire module.

[97deacc] Cutover to PlatformIO-only CI (v0.3.04)
  Dropped the build-arduino-cli and parity-check jobs after one release of
  verified parity. PlatformIO is now the single build. arduino-cli-legacy tag
  is preserved as a rollback anchor.

2. PlatformIO migration (v0.3.03)
────────────────────────────────────────────────────────────────────────────────

The migration from Arduino IDE / Arduino CLI to PlatformIO / VS Code hit a
string of interlocking issues. Lessons:

[53e1177] Switch to pioarduino platform (arduino-esp32 3.1.1)
  ROOT CAUSE: `espressif32@6.9.0` was assumed to ship arduino-esp32 3.1.1 but
  actually ships 2.0.17. The source code uses 3.x-only APIs
  (esp_now_recv_info_t, MBEDTLS_PRIVATE macros, NimBLE 2.x). PIO builds with
  the 2.0.17 core failed hard with:
    - mbedtls_ecp_point has no member named 'MBEDTLS_PRIVATE'
    - esp_now_recv_info_t does not name a type
  FIX: Switched platform to pioarduino's fork which tracks arduino-esp32 3.x:
    platform = https://github.com/pioarduino/platform-espressif32/releases/
               download/54.03.21/platform-espressif32.zip
  (The upstream platform-espressif32 repo was discontinued at 6.10.0.)

[72a0e5d] Pin click<8.2
  pioarduino 54.x uses an older click API. On click 8.2+ the bootloader.bin
  build step fails with "ParamType.get_metavar() missing 1 required
  positional argument: 'ctx'". Workaround:
    pip install --upgrade "platformio" "click<8.2"

[263d33b] compiler.cpp.extra_flags (not build.extra_flags)
  Critical gotcha specific to Arduino CLI. Using
    --build-property "build.extra_flags=-DFOO"
  REPLACES the arduino-esp32 platform's own flags (-DESP32, -DARDUINO_ARCH_ESP32,
  -DF_CPU=...). FastLED's platform auto-detection then falls back to AVR and
  fails with "This platform isn't recognized by FastLED... yet".
  FIX: Use `compiler.cpp.extra_flags` instead — it appends rather than replaces.

[ebc5626] Arduino IDE → PlatformIO directory restructure
  Repository layout change:
    esp32_firmware/esp32_firmware/esp32_firmware.ino → src/main.cpp
    esp32_firmware/esp32_firmware/*.h (22 headers)   → include/
    esp32_firmware/README.md                         → README.md (repo root)
    test/test_all.cpp + test_runner.h                → test/test_native/test_main.cpp
  Custom cl.exe test runner replaced by PlatformIO Unity framework:
    pio test -e native

[4e196da] Drop "-0.1.6" suffix from project folder name
  esp32_node_firmware-0.1.6/ → esp32_node_firmware/ (git renames preserved
  history). The suffix came from the initial zip download and was now stale.

[d0f8364] Restore min_spiffs + NimBLE
  Two regressions from the workflow rewrite:
    1. Arduino CLI failed with "text section exceeds available space in board"
       because I dropped --build-property "build.partitions=min_spiffs" +
       "upload.maximum_size=1966080" from the compile step. Restored both.
    2. PlatformIO failed with "NimBLEDevice.h: No such file or directory".
       NimBLE-Arduino is NOT bundled with arduino-esp32 3.x — it's a separate
       library that must be listed explicitly in lib_deps AND in the Arduino
       CLI install step.

4. Firmware correctness
────────────────────────────────────────────────────────────────────────────────

[71ba172] Dangling pointer for MQTT client ID and server host
  AsyncMqttClient::setClientId() and setServer() take raw `const char*`
  pointers and DO NOT copy the string. The original code passed `.c_str()`
  from temporary Strings that went out of scope. Fix: store the String at
  module scope (_mqttClientId, _mqttHost) so the pointer stays valid.

[d572283] Replace lexicographic OTA version compare with numeric semver
  ESP32-OTA-Pull's internal version comparison uses String::compareTo() which
  is lexicographic — "0.2.15" < "0.2.7" because '1' < '7'. Added a
  semverIsNewer() helper that parses the two strings as integers and compares
  MAJOR.MINOR.PATCH numerically.

[225304d] Tracked beacon stops updating when beacon list fills up
  BLE onResult callback had the tracked-beacon update logic INSIDE the guard
  that drops new beacons when _bleBeaconCount == BLE_MAX_BEACONS. Once 32
  unique MACs filled the list, the tracked beacon's seen-time stopped
  advancing and tracking silently died. Fix: moved the tracked-beacon update
  ABOVE the count guard.

[20bcd4d] Active scan for tracking, NimBLEAddress match
  Passive scans don't return scan responses → no manufacturer data → no
  iBeacon txPower. Fix: use active scanning for tracking cycles
  (pScan->setActiveScan(true) when durationS == BLE_TRACK_SCAN_DURATION_S).

[0b83b7e] BLE tracking scan must reset beacon list
  Tracking scans didn't reset _bleBeacons[] between scans, filling it with
  unique ephemeral MACs that pushed the tracked beacon's record out. Fix:
  clear _bleBeaconCount at the start of every scan, not just full scans.

5. Boot / release infrastructure
────────────────────────────────────────────────────────────────────────────────

[729fa78] Publish firmware + ota.json to GitHub Pages on release
  Initial OTA was using GitHub Releases URLs which 302-redirect to S3.
  ESP32-OTA-Pull's HTTP client didn't handle redirects cleanly. Fix: publish
  firmware.bin directly to the gh-pages branch so there's no redirect.

[d2f4a68] Restore config.h before gh-pages branch switch
  The CI's sed-patch of FIRMWARE_VERSION in config.h left the working tree
  dirty. `git checkout gh-pages` then refused to switch branches. Fix:
  `git checkout -- config.h` to revert the in-place edit before switching.
  (Obsolete in v0.3.03+ since FIRMWARE_VERSION is now injected via
  -DFIRMWARE_VERSION_OVERRIDE at compile time, not sed-patched.)

[bfb276e → 754a25b] Stash firmware.bin to /tmp before gh-pages checkout
  gh-pages already tracks firmware.bin; `git checkout` refuses to overwrite
  an untracked file of the same name. Solution: cp firmware.bin /tmp/ then
  rm the workspace copy before switching branches, cp back after.

[9f22... earlier] Workflow regen order
  Writing `git rm -rf . 2>/dev/null || true` is required on an orphan branch
  switch to clear the master-branch contents before writing just ota.json
  and firmware.bin. Without it, the gh-pages commit accumulates all source
  files.

6. BLE scanner
────────────────────────────────────────────────────────────────────────────────

[a23bb05] Switch BLE from Bluedroid to NimBLE
  Bluedroid adds ~700 KB to the firmware binary, overflowing the min_spiffs
  app partition (1.9 MB). NimBLE-Arduino adds ~150 KB — a 550 KB saving that
  fits comfortably.

[481f118] Update ble.h for NimBLE-Arduino 2.x API
  NimBLE 1.x → 2.x API changes that bit us:
    - Callbacks class renamed: NimBLEAdvertisedDeviceCallbacks → NimBLEScanCallbacks
    - onResult signature: now takes `const NimBLEAdvertisedDevice*` (pointer, const)
    - onScanEnd replaces the free-function callback previously passed to start()
    - pScan->start(durationS, is_continue): no longer takes a callback arg
    - Duration is in MILLISECONDS (1.x used seconds); multiply seconds * 1000

[225304d, 20bcd4d, 0b83b7e] See section 4 above.

7. OTA pipeline
────────────────────────────────────────────────────────────────────────────────

[d572283] Lexicographic compare → numeric semver (see section 4).

[ota.h] TaskWatchdog suppression during OTA
  During the OTA download, the async_tcp task holds a watchdog subscription
  but can't reset it because the TCP stack is saturated. Fix:
    TaskHandle_t asyncTcpTask = xTaskGetHandle("async_tcp");
    if (asyncTcpTask) esp_task_wdt_delete(asyncTcpTask);
  Safe because the device reboots immediately on OTA success.

[ota.h] MQTT disconnect before OTA
  OTA download blocks long enough that the AsyncMqttClient's reconnect timer
  fires and tries to reconnect mid-download, stalling async_tcp. Fix:
    _mqttClient.disconnect(true);
    delay(200);                 // let onMqttDisconnect fire + re-arm timer
    xTimerStop(_mqttReconnectTimer, 0);   // now kill the re-armed timer

8. MQTT and runtime behaviour
────────────────────────────────────────────────────────────────────────────────

[6ea91ef] Subscribe cmd/ota_check at QoS 0
  QoS 1/2 caused the broker to redeliver the cmd/ota_check message after the
  device restarted post-OTA, triggering an immediate second OTA check loop.
  QoS 0 prevents re-delivery — appropriate for fire-and-forget commands that
  trigger destructive actions (restart, OTA).

[21c035c] Subscribe cmd/restart at QoS 0 too
  Same issue — QoS 1 retry on reconnect → infinite restart loop.

[mqtt_client.h v0.3.00] Deferred restart flag replaces delay()
  Previously: mqttPublishStatus("restarting"); delay(100); ESP.restart();
  The delay inside an AsyncMqttClient callback stalls the keepalive loop and
  the client sometimes disconnected before the publish got out. Fix:
  set _mqttRestartAtMs = millis() + 300 and have mqttHeartbeat() check for
  it, actioning the ESP.restart() from the main loop context.

replace_espnow_flow.js / add_espnow_flow.js

11. NVS wear reduction (v0.3.05)
────────────────────────────────────────────────────────────────────────────────

[15c6565] Compare-before-write helper
  New include/nvs_utils.h — NvsPutIfChanged() overloads for uint8_t,
  uint16_t, uint64_t, null-terminated strings, and raw bytes. Each overload
  reads the existing NVS value and returns success (without writing) when it
  matches. Return semantics mimic Arduino's Preferences::put*() so existing
  `ok &= prefs.putX(...)` accumulator chains keep working.

  Migrated call-sites:
    - credentials.h        CredentialStore::save()  + clearRestartCount()
    - broker_discovery.h   saveBrokerToCache()
    - rfid.h               _rfidWhitelistSave()
    - app_config.h         AppConfigStore::save()
    - ws2812.h             _ledNvsSave()

  Real-world impact:
    - Node-RED slider at rest → no NVS write on repeat cmd/led brightness
    - OTA boot with unchanged broker cache → no write
    - Admin re-saving settings portal with 1 field changed → 1 write, not 7
    - Credential rotation replay of identical bundle → no writes at all

  Unit tests: 8 Unity assertions against a FakePrefs harness in
  test/test_native/test_main.cpp (27 total tests, all pass in `pio test`).

12. Structured logging + FwEvent enum (v0.3.06)
────────────────────────────────────────────────────────────────────────────────

New files: include/logging.h, include/fwevent.h

[logging.h] Compile-time filtered LOG_D/I/W/E macros
  214 Serial.printf / Serial.println call-sites across mqtt_client.h, ota.h,
  and main.cpp converted to LOG_*(tag, fmt, ...) macros. Output format changes
  from "[MQTT] message" to "[I][MQTT] message" (level prefix added).
  Level filtering is controlled by LOG_LEVEL in config.h (default: INFO).
  Build with -DLOG_LEVEL=LOG_LEVEL_WARN to suppress informational output in
  production without reflashing — zero runtime cost for excluded levels.
  The tag + fmt arguments are string literals concatenated at compile time;
  no heap allocation occurs.

[fwevent.h] FwEvent enum replaces free-form mqttPublishStatus() string literals
  Ten event strings ("boot", "heartbeat", "ota_checking", ...) promoted to
  enum class FwEvent : uint8_t. fwEventName() provides the string lookup.
  A typed overload mqttPublishStatus(FwEvent, const char*) added in mqtt_client.h
  delegates to the existing const char* version so the retain logic
  (strcmp(event, "boot")) and the Node-RED string contract are both preserved.
  fwevent.h has no Arduino/FreeRTOS dependencies and can be included in host-side
  Unity tests directly.

[main.cpp] Non-blocking loop() Wi-Fi recovery
  Replaced the blocking while/delay(3000) reconnect loop with a static-state
  pattern: _wifiRecoveryAttempts and _wifiRetryAt track progress across loop()
  iterations. loop() returns after each failed attempt, allowing MQTT heartbeat,
  OTA, and RFID/BLE tasks to continue running during the 3-second inter-attempt
  pause. connectWifi() itself still blocks for up to WIFI_CONNECT_TIMEOUT_MS —
  full non-blocking WiFi recovery is deferred to v0.3.07.

[test_main.cpp] 3 new Unity tests for fwEventName() (30 total)
  test_fwevent_boot_is_retained_string — guards the "boot" string that onMqttConnect
    relies on for its retain-flag strcmp check.
  test_fwevent_all_known_values — asserts every enum value maps to its expected
    Node-RED contract string.
  test_fwevent_unknown_value_returns_string — confirms the default branch never
    returns nullptr (which would crash the JSON builder).

13. v0.3.08 — Wire-format versioning, snprintf, NVS namespace registry, tests
────────────────────────────────────────────────────────────────────────────────

[espnow_bootstrap.h] WireBundle wire_version field (DEFERRED item 11)
  WireBundle gained a uint8_t wire_version as its first byte (struct size 175→176).
  ESPNOW_WIRE_VERSION constant = 1. bundleToWire() sets wire_version = 1.
  Both receive paths (espnowBootstrap and espnowBootstrapWithPrimarySelection)
  check wire.wire_version after decryption and LOG_W + reject frames with any
  value other than 1. Old senders (pre-v0.3.08) send 175-byte plaintext which
  fails the size check first — they are rejected with "Bundle size mismatch".
  logging.h included in espnow_bootstrap.h for the new LOG_W calls.

[ap_portal.h] apHandleStatus snprintf (DEFERRED item 12)
  apHandleStatus() replaced String concatenation (6 heap allocs per call) with
  a single 512-byte stack buffer + snprintf(). No behaviour change — field values
  and JSON structure identical.

[config.h] NvsNs enum registry (DEFERRED item 13)
  enum class NvsNs : uint8_t added after the BLE namespace #define, with values
  CREDENTIALS(0), DEVICE_ID(1), BROKER_CACHE(2), LED(3), RFID(4), BLE(5).
  inline nvsNsName(NvsNs) returns the string from the existing #define so
  all six existing macro call-sites compile unchanged.

[include/semver.h] New dependency-free header (DEFERRED item 14)
  semverIsNewer() extracted from ota.h into include/semver.h which only
  #includes <stdio.h>. ota.h now #includes "semver.h". The function is
  unchanged — refactor only.

[test_main.cpp] 10 new Unity tests (40 total) (DEFERRED item 14)
  6 semver tests: major/minor ordering, double-digit minor (0.9.x < 0.10.x),
  numeric patch comparison (0.2.7 < 0.2.15), leading zeros (0.3.00 == 0.3.0),
  equal versions.
  4 topic sanitizer pathological cases: multiple slashes in one segment,
  leading slash, segment consisting only of wildcards, mixed wildcard + text.

14. v0.3.09 — Non-blocking ESP-NOW scan + AsyncTCP parallel broker port scan
────────────────────────────────────────────────────────────────────────────────

[espnow_bootstrap.h] Async FreeRTOS task API (DEFERRED items 4, 6)
  Added espnowBootstrapBegin() / espnowBootstrapIsDone() / espnowBootstrapGetResult()
  backed by a FreeRTOS task (_espnowBootstrapTask, 8 KB stack, priority 5).
  The task runs the existing espnowBootstrap() BOOTSTRAP_MAX_ATTEMPTS times
  with vTaskDelay(2000 ms) between retries and self-deletes on completion.
  setup() calls espnowBootstrapBegin() and returns immediately; loop() polls
  espnowBootstrapIsDone() in the BOOTSTRAP_REQUEST state.

[broker_discovery.h] AsyncTCP parallel port scan (DEFERRED items 5)
  tryPortScan() replaced with an async sliding-window implementation using
  AsyncTCP. PORTSCAN_WINDOW = PORTSCAN_MAX_RESULTS + 2 = 6 parallel AsyncClient
  connections. Each slot fires a connect(); callbacks set volatile done/open
  flags; main task polls with delay(5) to yield to the AsyncTCP/lwIP FreeRTOS
  task. Manual per-slot timeout at 2 × PORTSCAN_TIMEOUT_MS (150 ms) prevents
  stalls. Worst-case time: ceil(254/6) × ~180 ms ≈ 7.6 s vs 19 s synchronous.
  #include <AsyncTCP.h> and #include "logging.h" added; all Serial.printf/
  println calls in the file converted to LOG_I/LOG_W macros.

[src/main.cpp] Non-blocking boot state machine in loop() (DEFERRED items 4, 5)
  setup() now returns immediately after spawning the bootstrap task (or calling
  WiFi.begin() for the fast re-boot path). loop() drives the full state machine:
  — BOOTSTRAP_REQUEST: polls espnowBootstrapIsDone(); on completion processes the
    bundle (timestamp cap, NVS compare) and transitions to WIFI_CONNECT.
  — WIFI_CONNECT: non-blocking WiFi.begin() + millis() deadline polling; retries
    up to WIFI_MAX_ATTEMPTS; synchronous espnowBootstrap() sibling re-verify on
    total failure (SIBLING_REVERIFY_ATTEMPTS == 1, rare path).
  — AP_MODE: calls apPortalStart() (never returns).
  — OPERATIONAL: one-time service startup on first entry (espnowResponderStart,
    discoverBroker, mqttBegin, rfidInit, bleInit); subsequent iterations run
    non-blocking Wi-Fi recovery + MQTT watchdog + OTA + ranging + RFID + BLE.
  WiFiEvent registered in setup() before any WiFi.begin() call.
  All remaining TODO(v0.3.07) comments removed.

  be read from outside ap_portal.h.

[rfid.h] Module-local state flag accessor (DEFERRED item 9)
  _rfidReady kept as a static module-private flag. An inline rfidIsReady()
  accessor added with a matching doc comment.

[espnow_responder.h] Module-local state flag accessor (DEFERRED item 9)
  _responderActive kept as a static module-private flag. An inline
  espnowResponderIsActive() accessor added.

[espnow_responder.h] Per-peer token-bucket rate limiter (DEFERRED item 10)
  Replaced the previous flat 30-second per-MAC cooldown with a proper token-
  bucket table: RATE_BUCKET_CAP=3 tokens, RATE_REFILL_MS=60000 (1 per minute),
  RATE_MAC_SLOTS=8 entries. Each credential request consumes 1 token; refill
  computed as floor(elapsed / RATE_REFILL_MS) capped at RATE_BUCKET_CAP. New
  MACs start at CAP-1 so the very first request always passes. The table uses a
  circular _rateBucketNext eviction index (LRU-approximate). Requests exceeding
  the budget are silently dropped with LOG_W("Responder", "rate-limit dropped").

[espnow_responder.h, broker_discovery.h] LOG_* macro adoption
  All remaining Serial.printf / Serial.println calls in both files converted to
  LOG_I / LOG_W / LOG_E macros (introduced in v0.3.06). Both files now include
  "logging.h". espnow_responder.h covers all handlers: onEspNowRequest,
  onEspNowOtaUrlRequest/Response, onEspNowBrokerRequest/Response,
  espnowGetSiblingBroker, and espnowResponderStart.


────────────────────────────────────────────────────────────────────────────────
MISC REMINDERS
────────────────────────────────────────────────────────────────────────────────

• Every CI debug iteration costs ~2-3 minutes of runner time. When CI fails,
  read the FULL log (especially pre-fail warnings) before pushing a speculative
  fix. Most of the v0.3.03 fix loop could have been collapsed into 1-2 commits
  if I'd re-checked the v0.3.02 WORKING build.yml before making assumptions.

• The Arduino CLI library registry is NOT a stable dependency source.
  If a library install fails mysteriously ("latest not found"), switch to
  cloning from GitHub. This applies to MFRC522v2, NimBLE-Arduino (when
  multiple registries disagree), and has bitten us on the marvinroger/
  me-no-dev/mikalhart trio from the start.

• `defaults.run.working-directory:` in a workflow applies to every `run:`
  step. Actions (checkout, upload-artifact, download-artifact) get paths
  relative to the checkout root and need the subfolder prefix explicitly.

• pioarduino releases its platform zips under
  https://github.com/pioarduino/platform-espressif32/releases — check there
  for new versions if arduino-esp32 upstream ships a major core update.

• The FIRMWARE_VERSION macro in config.h is now set by CI via the
  -DFIRMWARE_VERSION_OVERRIDE build flag. Local dev builds fall through to
  "0.3.05-dev" or similar. Do NOT bump the fallback literal for each release
  — only bump it when starting work on the NEXT version, so local dev can
  distinguish dev binaries from the previously released one.

• `arduino-cli-legacy` tag points to the last commit where the Arduino CLI
  workflow was still the canonical build path (post v0.3.00, pre-migration).
  Useful rollback anchor if the PlatformIO stack breaks badly.
